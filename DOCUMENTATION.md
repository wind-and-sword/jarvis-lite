# 项目文档整理约定

> 日期：2026-05-18
> 执行者：Codex
> 适用范围：Jarvis Lite 项目内的正式文档、过程记录和临时日志。

## 结论

后续本项目内的过程性文档、方案文档、内部工作文档和临时日志，默认按本文件约定执行。

这样做有两个直接目标：

1. 方便用户随时阅读当前执行进度和历史方案。
2. 方便 Codex 快速检索以往记录，判断下一步应从哪里继续执行。

## 具体规则

1. 先区分文档类型，再决定放置目录和命名方式，不能把方案文档、进度记录、内部工作文档和临时日志混放。
2. `DOCUMENTATION.md` 是项目级长期文档约定，保留在根目录并提交 Git；不要迁移到 `.codex/`，避免被本地忽略规则隐藏。
3. 面向用户阅读的正式文档只放在 `word/` 目录，默认使用 Markdown，也就是 `.md` 文件；除非用户明确要求，不生成 `.doc`、`.docx` 或 `output/doc/` 这类额外交付目录。
4. 方案设计文档放在 `word/` 目录，文件名以主题为主，不强制带日期，例如 `jarvis-lite-phase-1.md`、`jarvis-lite-memory-design.md`。
5. 执行进度记录也放在 `word/` 目录，文件名必须带日期前缀，统一使用 `YYYY-MM-DD-主题-progress.md`。
6. 根目录下的 `README.md` 是项目入口说明，保留在根目录；根目录下的 `日志.txt` 只作为临时运行日志或排障日志，不作为正式方案文档或正式进度文档。
7. `日志.txt` 可以随时清空；其中有价值的结论必须整理进 `.codex/operations-log.md` 或 `word/*progress.md`，不能只留在临时日志中。
8. 仅供 Codex 自己使用的临时上下文、计划、验证和操作记录统一放在 `.codex/` 目录，不放在 `docs/` 目录，也不放在用户常读的文档目录中。
9. `.codex/` 默认是本地工作目录，不作为长期共享文档的唯一来源；需要提交到 GitHub 或长期保留的规则、方案、结论，应整理到根目录或 `word/`。
10. 同一事项需要留两份时，给用户看的版本放 `word/`，给 Codex 自己检索和复盘的版本放 `.codex/`，两份文件不能混写。
11. 内部工作文档默认使用中文编写，除非确实需要保留英文术语或命令原文。
12. 面向用户阅读的文档默认使用中文，尽量直接说明“当前做到哪一步”“下一步做什么”“哪些事项待确认”。
13. 首次创建 `word/` 正式文档时，同步创建 `word/文档索引.md`；后续新增或调整 `word/` 下的文档时，要同步更新索引，保证文档入口始终清晰。
14. 如果某个主题同时存在“方案设计”和“执行进度”两类文档，必须通过文件名和索引明确区分，避免混淆。
15. 同一主题下，如执行进度发生明显阶段变化，优先新建带日期的新进度文档，而不是长期覆盖同一个无日期文件。

## 当前目录分工

- 根目录：项目入口说明、构建文件、项目级约定和临时日志；不作为专题方案文档主目录。
- `word/`：给用户和开发过程共同查阅的正式方案文档、设计文档、进度记录，默认使用 `.md`。
- `.codex/`：给 Codex 自己使用的本地上下文、计划、测试、验证和操作记录。
- `logs/`：后续助手运行日志目录，不作为文档归档入口；重要结论需要整理到 `word/` 或 `.codex/`。

## 推荐示例

```text
README.md
DOCUMENTATION.md
日志.txt
word/jarvis-lite-phase-1.md
word/2026-05-18-jarvis-lite-progress.md
word/文档索引.md
.codex/context-scan.json
.codex/operations-log.md
.codex/testing.md
```

## 备注

如果后续某类文档命名或归档方式需要调整，优先先改本文件，再批量调整具体文档，避免再次出现口径不一致。
