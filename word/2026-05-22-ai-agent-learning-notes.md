## 1. 这次讨论解决的问题

这份记录用于沉淀个人 AI 技能地图，重点不是背概念，而是把 Agent 放到工程系统里理解。

本次讨论逐步澄清了几个核心问题：

- Agent 不是普通聊天机器人，而是能调用工具完成任务的 AI 执行系统。
- Agent 不是 LLM 后面的一个模块；Agent 本身包含 LLM 调用。
- RAG 本身不思考，它负责检索资料；真正判断和总结通常由 Agent 内部的 LLM 完成。
- AI 应用开发和传统业务系统开发有连续性，但多了一个非确定性决策层。
- 对 Java 后端来说，最重要的不是训练模型，而是把 AI 接入已有业务系统和工具。

## 2. Agent 的工程化定义

一句话：

> Agent 是一个围绕 LLM 构建、能够使用工具、管理状态并完成任务的执行系统。

更准确的结构是：

```text
Agent
  ├── LLM：理解、规划、判断、总结
  ├── Tool Calling：调用函数、接口、数据库、脚本
  ├── RAG：检索内部知识库、文档、规则
  ├── Memory：保存长期偏好、短期上下文、经验
  ├── Workflow：控制任务步骤和执行顺序
  ├── State：保存中间结果和待确认动作
  └── Tools / API / DB / 文件系统：真正执行动作的外部世界
```

所以不能画成：

```text
LLM -> Agent
```

更应该画成：

```text
用户
  ↓
AI 应用框架
  ↓
Agent
  ├── LLM
  ├── Memory
  ├── RAG
  ├── Tools
  └── Workflow
```

## 3. 普通 Chat 和 Agent 的区别

普通 Chat：

```text
用户：帮我写一个 SQL
AI：返回一段 SQL
结束
```

Agent：

```text
用户：帮我分析今天广州订单异常原因
Agent：
  1. 理解任务
  2. 查询订单数据库
  3. 查询支付失败日志
  4. 检索内部故障手册
  5. 汇总异常原因
  6. 生成处理建议
  7. 必要时通知相关人员
```

关键差异不是“会不会回答”，而是“会不会行动”。

## 4. AI 应用和传统业务系统的关系

用户提出的判断是：AI 应用开发和传统业务系统开发本质上很像，都是围绕业务能力做组合。

这个判断大体成立，但需要补充一个差异：

> 传统系统是确定性调用链，AI 应用多了非确定性决策层。

传统业务系统：

```text
按钮
  ↓
固定接口
  ↓
固定 Service
  ↓
固定 SQL / 固定流程
```

AI 应用：

```text
自然语言目标
  ↓
Agent 理解意图
  ↓
Agent 决定调用哪些工具
  ↓
工具返回结果
  ↓
Agent 组织结论
```

传统系统里，程序员提前写死流程。Agent 系统里，程序员提供工具和边界，Agent 动态选择流程。

## 5. RAG 的准确位置

RAG 是 Retrieval-Augmented Generation，检索增强生成。

它的本质是：

```text
先查资料，再让 LLM 基于资料回答
```

RAG 自身通常没有思考能力。它更像资料员，负责从知识库中找候选内容。

典型流程：

```text
用户问题
  ↓
Agent 调 LLM 理解任务
  ↓
Agent 判断需要 RAG
  ↓
RAG 检索候选资料
  ↓
排序模型 / 规则 / LLM 筛选资料
  ↓
Agent 调 LLM 基于资料推理
  ↓
返回结论
```

如果 RAG 返回很多文档，通常不会全部塞给 LLM。真实系统会先做筛选：

- 关键词检索或向量检索先找出候选。
- rerank 模型或规则挑出最相关片段。
- 只有少量高相关证据进入最终 LLM 上下文。

一句话：

> RAG 是资料员，LLM 是分析师，Agent 是把资料员、分析师和业务工具组织起来干活的负责人。

## 6. Agent 是否每一步都调用 LLM

不是。

Agent 包含 LLM，所以它可以在多个关键节点调用 LLM；但成熟系统不会每一步都问大模型。

更合理的分工是：

```text
用户意图理解：通常调用 LLM
参数抽取：可用 LLM，也可用规则
是否需要 RAG：可用 LLM，也可用规则
RAG 检索：通常不需要 LLM
候选资料排序：可用专门 rerank 模型，也可用规则或 LLM
数据库/API 调用：代码执行，不需要 LLM
最终分析总结：通常调用 LLM
结果格式化和落库：代码执行
```

原则：

> 理解、决策、推理、总结交给 LLM；确定性执行交给代码、规则和工具。

## 7. Java 开发者的学习重点

这次讨论形成的学习优先级：

1. Function Calling：让模型决定调用哪个函数、传什么参数。
2. RAG：让 AI 读取企业内部知识库。
3. Spring AI 或 LangChain4j：在 Java/Spring Boot 中组织模型、工具、RAG、上下文。
4. MCP：用统一协议接入数据库、Git、文件系统、内部业务系统等工具。
5. Agent Workflow：设计多步骤任务执行、状态管理和人工确认点。

不优先投入：

- Transformer 数学推导。
- CUDA 和训练框架。
- 大模型预训练。
- 盲目追多 Agent 协作。

当前最有价值的路线是：

> 先把 AI 调接口、AI 查知识库、AI 接业务系统跑通。

## 8. 对 Jarvis Lite 的启发

企业 Agent 是让 AI 使用企业系统。

Jarvis Lite 的方向是：

> 让 AI 使用个人设备和个人数据。

企业场景里的工具是：

```text
订单系统 / 支付系统 / CRM / OA / 日志系统 / 数据库
```

Jarvis Lite 场景里的工具是：

```text
电脑文件系统 / 桌面应用 / 浏览器 / 日历 / 邮件 / 手机通知 / 手表健康数据 / 本地知识库
```

因此 Jarvis Lite 可以理解为个人设备级 Agent 的雏形。

## 9. 当前认知结论

- Agent 是任务执行系统，不是单纯聊天。
- Agent 包含 LLM，但不等于 LLM。
- RAG 是检索能力，不是思考能力。
- Function Calling 是模型和业务系统之间的桥。
- MCP 是工具接入协议，不是 Agent 本身。
- 企业 AI 应用的核心价值是让 AI 使用已有系统。
- Jarvis Lite 的长期价值是让 AI 使用个人设备和个人数字生活。
