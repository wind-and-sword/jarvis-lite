# Jarvis Lite 桌面虚拟助手应用方案

> 日期：2026-05-19
> 执行者：Codex

## 1. 结论

下一阶段应把 Jarvis Lite 从命令行助手推进为桌面虚拟助手应用。推荐形态是“两层体验”：平时桌面角落常驻一个小型虚拟助手角色，点击后展开完整助手面板。

这条路线比传统大窗口应用更符合“个人智能助手”的感觉：它不是一个需要主动打开的软件，而是一个能一直在旁边待命、反馈状态、承接对话和工具动作的本地伙伴。

第一版仍然不接入摄像头、麦克风和真实硬件入口。当前重点是把已有命令行能力包装成可视化、可长期停留在桌面的应用体验。

## 2. 当前基础

目前已经具备适合做桌面应用后端的核心能力：

- `JarvisAgent.handle(user_input)` 可以处理单轮输入和命令。
- `ConversationSession.handle(user_input)` 可以管理多轮会话、历史和会话总结。
- 长期记忆、资料问答、知识库导入、标签、日报、常用目录、文件整理预览和目录打开记录都已经在 Python 核心内实现。
- 阶段 4 自动化目前都是低风险动作，适合先接入 UI。
- 摄像头、麦克风和真实外部应用启动已按用户要求暂缓。

因此，下一步不应该重写助手核心，而应该新增一个桌面应用壳，让 UI 调用现有核心。

## 3. 产品形态

### 3.1 桌面小助手

桌面上常驻一个小角色，默认停在屏幕角落，可以拖动位置。它表达 Jarvis Lite 的状态：

- 待机：没有任务，安静停留。
- 思考：正在处理用户输入。
- 工作：正在执行命令，例如生成日报、检索知识库。
- 完成：任务成功。
- 提醒：有可读结果或需要用户查看。
- 错误：命令失败或输入不完整。

第一版不追求复杂养成系统，不做小游戏化数值，不做宠物喂养。角色的目标是“存在感”和“反馈感”，不是抢走助手主线。

### 3.2 助手面板

点击桌面小助手后展开完整面板。第一版面板包含：

- 对话区：输入文本，展示用户和助手回复。
- 快捷命令区：一键触发 `/status`、`/kb`、`/dirs`、`/daily-report`、`/organize-preview` 等能力。
- 状态区：显示长期记忆、知识库、自动化能力和最近工具日志摘要。
- 设置区：控制置顶、窗口大小、角色状态显示和 transcript 模式。

面板不做复杂管理后台。第一版目标是让用户不再需要打开命令行，也能完成当前已有能力。

### 3.3 交互原则

- 平时尽量小，不挡工作。
- 需要输入时展开，不让用户记命令。
- 工具动作必须有可见状态反馈。
- 高风险能力后续必须有确认流程；当前第一版只接低风险或 dry-run 能力。
- UI 文案保持直接，不写大段说明。

## 4. 技术方案

### 4.1 推荐方案：PySide6 桌面壳 + 现有 Python 核心

推荐第一版使用 PySide6/Qt 构建桌面应用。原因：

- 当前核心代码全部是 Python，可以直接调用 `JarvisAgent` 和 `ConversationSession`。
- 可以做透明、无边框、置顶的小窗口，适合桌面虚拟助手形态。
- 可以做系统托盘、快捷键、设置窗口和本地文件访问。
- 不需要第一版就引入 Node、Rust、Web sidecar 或多进程通信。
- 测试可以沿用 Python 本地测试，GUI 层再增加最小 smoke 验证。

### 4.2 备选方案

方案 A：本地 Web UI。

- 优点：开发快，界面自由度高。
- 缺点：不是真正桌面宠物，需要浏览器承载，桌面存在感弱。

方案 B：Tauri 或 Electron。

- 优点：更接近现代桌面应用，前端生态强。
- 缺点：需要额外处理 Python sidecar、打包、进程通信和更多工具链，第一版维护面偏大。

方案 C：PySide6。

- 优点：与当前 Python 核心最贴合，能快速做桌面小助手和面板。
- 缺点：视觉和动效不如 Web 前端自由，后续复杂动画可能需要更多资产设计。

综合判断：第一版先用 PySide6。等核心应用体验稳定后，再决定是否迁移到 Tauri 或引入 WebView。

## 5. 模块边界

建议新增桌面模块，但不改动现有核心职责：

```text
src/jarvis_lite/
  agent.py                 继续负责命令和工具调度
  conversation.py          继续负责会话状态
  desktop/
    app.py                 桌面应用入口
    bridge.py              UI 到 ConversationSession 的适配层
    state.py               助手状态：idle、thinking、working、success、error
    widgets.py             桌面角色窗口和助手面板
    settings.py            读取和保存 UI 设置
    assets/                角色图片、状态图和图标
```

`desktop/bridge.py` 是关键边界。UI 不直接散落调用各个工具函数，而是统一通过 bridge 调用现有 `ConversationSession`。这样后续更换 UI 技术时，助手核心仍然稳定。

## 6. 数据流

第一版数据流如下：

1. 用户点击桌面小助手，展开面板。
2. 用户输入文本或点击快捷命令。
3. UI 把输入交给 `DesktopBridge.send(text)`。
4. `DesktopBridge` 调用 `ConversationSession.handle(text)`。
5. 核心返回助手文本。
6. UI 更新对话记录和角色状态。
7. 工具调用继续写入现有 `logs/`。

状态映射规则：

- 输入发送后：`thinking`
- 命令类输入：`working`
- 返回正常结果：`success`
- 返回用法错误、失败或异常：`error`
- 一段时间无动作：`idle`

## 7. 第一版范围

第一版桌面应用只做这些：

- 启动一个桌面小助手窗口。
- 支持拖动位置和置顶显示。
- 点击小助手展开/收起助手面板。
- 面板支持文本输入和显示对话历史。
- 面板支持几个快捷命令按钮。
- 复用现有长期记忆、知识库和自动化能力。
- 角色能根据 idle、thinking、working、success、error 切换状态。
- UI 设置保存到本地文件。
- 保留命令行入口，不影响现有测试。

第一版不做这些：

- 不接入摄像头。
- 不接入真实麦克风识别。
- 不做真实外部应用启动。
- 不做宠物养成数值系统。
- 不做联网大模型 API。
- 不做跨平台安装包。

## 8. 视觉方向

视觉上应该偏“安静的桌面助手”，不是花哨游戏界面。

建议第一版使用 5 张角色状态图或一张 sprite sheet：

- idle
- thinking
- working
- success
- error

角色尺寸建议默认 160 到 220 像素。面板尺寸建议默认 420 x 620 像素。面板应像工具窗口，信息密度适中，方便反复使用。

角色资产可以先使用本地生成的 PNG 图，后续再统一风格。不要用纯文字方块代替角色，否则不像虚拟助手。

## 9. 验收标准

桌面应用第一版完成时，应满足：

- 可以通过命令启动桌面应用。
- 桌面上出现可拖动的小助手。
- 点击小助手可以展开和收起助手面板。
- 在面板输入“你知道我是谁吗”可以复用长期记忆回答。
- 在面板输入 `/kb` 可以查看知识库状态。
- 点击快捷命令可以触发日报、目录列表或整理预览。
- 执行命令时角色状态会变化。
- 命令行入口仍然可用。
- 本地测试通过，不依赖 CI。

## 10. 实施顺序

建议分 5 步推进：

1. 新增桌面 bridge 层，把 `ConversationSession` 包装成 UI 可调用接口。
2. 引入 PySide6，创建最小桌面窗口和助手面板。
3. 增加桌面小助手窗口：置顶、拖动、点击展开面板。
4. 接入对话输入、快捷命令和状态切换。
5. 增加角色图片资产、设置保存和文档。

每一步都保持命令行能力可用，并补充对应本地测试或 smoke 验证。

## 11. 风险

- PySide6 依赖体积较大，后续打包会增加应用体积。
- Windows 透明置顶窗口和拖动行为需要实际截图验证。
- 角色资产会影响第一印象，需要单独打磨。
- 如果后续改用 Tauri，需要重新设计 UI 与 Python 核心通信方式。
- 当前助手回答仍然是规则式，不接入大模型时，智能感主要来自记忆、知识库和工具执行。

## 12. 下一步
