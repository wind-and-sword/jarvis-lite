# Jarvis Lite v4：内脑与外脑双脑架构方案

> 日期：2026-05-28
> 执行者：Codex
> 说明：本文承接 v3 的 PC Agent 与 LLM 外脑优先路线，进一步明确自然语言理解不应继续堆正则，而应引入本地内脑 InnerBrain，与 LLM 外脑形成分工。

## 1. 背景

用户在安装 `0.1.1` 后的真实聊天日志暴露了一个关键问题：Jarvis Lite 已经具备命令、知识库、桌面和 LLM fallback，但自然语言入口仍然依赖有限规则。补几条正则可以修复具体样例，却不能解决长期问题。

继续把用户可能说的每句话都写成正则是不合理的。Jarvis Lite 需要一个本地、轻量、可训练、可评估的自然语言内脑，用来把用户表达转换为可执行的结构化任务。

## 2. 核心判断

Jarvis Lite 不应该从零训练一个小型通用 LLM。通用语言模型需要大量高质量语料、算力、评测和长期维护，不适合作为当前项目的本地核心。

更合理的内脑是一个小型 NLU 与决策模型：

- 识别用户意图。
- 抽取关键参数。
- 输出置信度。
- 判断是否缺少信息。
- 决定直接执行、追问确认，还是交给 LLM 外脑。
- 对简单场景生成基础回复，或把结构化结果交给回复生成层。

## 3. 双脑分工

```text
用户自然语言
  -> InnerBrain 本地内脑
     -> intent：用户想做什么
     -> slots：对象、路径、文件名、时间、数量等
     -> confidence：识别置信度
     -> missing：缺失信息
     -> action_policy：执行 / 追问 / 交给外脑
  -> JarvisAgent 决策与执行层
     -> 高置信度且低风险：直接执行
     -> 中置信度或信息缺失：追问澄清
     -> 低置信度或开放问题：交给 LLMRouter
  -> 工具执行 / 上下文更新 / 结果反馈
```

### InnerBrain 内脑

内脑负责本地、快速、可控的自然语言理解，不直接追求开放聊天能力。

职责：

- 将自然语言映射为 `intent + slots + confidence`。
- 学习用户真实表达和项目已有命令之间的映射。
- 处理高频本地任务：知识库、最近上下文、目录、桌面、更新、记忆、经验、文件操作。
- 在信息不足时生成澄清问题。
- 把错误识别样本沉淀为训练数据。

不负责：

- 长篇自由问答。
- 复杂推理和开放知识总结。
- 直接绕过 `JarvisAgent` 执行工具。

### LLM 外脑

LLM 外脑继续负责复杂理解、总结、规划和自然表达。

职责：

- 复杂、多轮、开放式自然语言理解。
- 资料总结、任务拆解、经验归纳。
- 为不确定输入生成候选结构化意图或澄清问题。
- 帮助扩展训练数据和表达变体。

不负责：

- 直接执行系统动作。
- 绕过本地命令白名单。
- 保存或暴露真实 API key。

## 4. 数据与训练策略

不建议从百度、谷歌等搜索结果直接抓取无标签自然语言来训练。搜索结果通常没有 Jarvis 意图标签，噪声高，和本地 PC Agent 任务映射弱，也存在来源和版权边界问题。

Jarvis Lite 需要的是带标签的任务样本：

```json
{
  "text": "帮我把桌面上的比特浏览器快捷方式删掉",
  "intent": "desktop.delete_shortcut",
  "slots": {
    "location": "desktop",
    "items": ["比特浏览器"]
  }
}
```

推荐数据来源：

- 项目已有命令和能力定义。
- 用户真实聊天日志中识别失败或识别正确的样本。
- LLM 外脑按既有 intent schema 生成的中文表达变体。
- 每次用户纠正后的 `text -> intent -> slots` 样本。
- 自动化测试中固化的回归样例。

## 5. 第一版实现建议

第一版内脑不做端到端深度学习模型，先做可控、可验证的轻量方案：

```text
训练样本库
  -> 文本归一化
  -> 字符 n-gram / 轻量 embedding 特征
  -> 意图分类器或相似度检索
  -> 槽位抽取
  -> 置信度与策略判断
```

推荐阶段：

1. 定义统一的 `IntentResult`：
   - `intent`
   - `slots`
   - `confidence`
   - `missing`
   - `source`
   - `reason`
2. 建立 `data/inner-brain/training/*.jsonl` 样本格式。
3. 用现有自然语言规则和命令能力生成第一批种子样本。
4. 用 LLM 外脑批量生成表达变体，但必须落到固定 schema。
5. 训练或构建一个本地轻量分类器。
6. 在 `JarvisAgent` 前增加 `InnerBrain` 判断层。
7. 低置信度继续走 LLM fallback。
8. 把用户纠错和失败样例写入待标注数据。

## 6. 回复行为原则

自然语言回复不应全部固定模板，也不应全部交给 LLM 自由生成。

推荐规则：

- 执行结果、错误、路径、版本号、删除/写入/导入结果：结构化固定，保证可验证。
- 问候、身份、能力说明、上下文解释、建议：允许更自然的回复。
- 高风险操作：先说明识别结果和影响范围，必要时追问确认。
- 低置信度识别：不要装作理解，返回澄清问题或交给外脑。
- LLM 生成的自然回复不得改变工具实际执行结果。

## 7. 与现有系统的关系

v4 不推翻 v3，而是在 v3 的 LLM 外脑路线前增加一个更稳定的本地理解层：

```text
命令 / 显式操作
  -> JarvisAgent

自然语言
  -> InnerBrain
  -> JarvisAgent
  -> LLMRouter fallback
```

现有 `parse_natural_language_intent()` 可以作为 InnerBrain 第一阶段的兼容层逐步迁移；新增内脑后，不再把长期方向定义为继续堆正则。

## 8. 验收标准

- 用户真实日志中的失败样例会进入训练或回归样本。
- 每个 intent 都有明确 schema 和测试样例。
- 内脑输出必须包含置信度和来源，便于调试。
- 高风险动作不会因为低置信度被直接执行。
- LLM 外脑只负责理解增强和表达增强，执行仍由 `JarvisAgent` 控制。
- 全量本地测试、桌面 smoke 和文档链接检查通过。

## 9. 暂不做

- 不从搜索引擎直接抓取无标签语料训练。
- 不从零训练通用小型 LLM。
- 不让 LLM 直接调用系统工具。
- 不为了自然回复牺牲执行结果的可验证性。
