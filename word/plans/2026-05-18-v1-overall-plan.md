# Jarvis Lite 整体方案与路线图

> 日期：2026-05-18
> 执行者：Codex

> 历史版本说明：本文是 v1 初版方案，保留用于追溯。当前方案以 [../PROJECT-PLAN.md](../PROJECT-PLAN.md) 和 [2026-05-27-v3-pc-agent-llm-first-plan.md](2026-05-27-v3-pc-agent-llm-first-plan.md) 为准。

## 1. 项目起点

本项目源于对“贾维斯式个人智能助手”的讨论。我们区分了两个目标：

- 电影版贾维斯：长期在线、深度接入现实设备、具备高度自主判断和稳定通用智能。这个目标目前不能一步到位。
- 实用版贾维斯：能语音或文本对话，理解个人偏好，调用工具做事，管理文件、知识、日程、代码和自动化流程。这个目标可以从最小版本逐步实现。

核心判断是：所有“不可能”的东西都应从最小可运行版本开始，而不是停留在想象中。

## 2. 项目愿景

Jarvis Lite 是一个属于个人的本地智能助手起点。它不是单纯聊天机器人，而是一个能逐步接入真实工作和生活流程的本地智能中枢。

长期方向包括：

- 能和用户自然对话，理解上下文和个人表达习惯。
- 能记住长期偏好、项目背景、重要事件和常用规则。
- 能调用本机工具，帮助处理文件、代码、文档、搜索和自动化任务。
- 能根据权限边界执行动作：低风险任务自动执行，高风险任务先确认。
- 能逐步接入语音、桌面、手机、家居、车机等外部入口。

## 3. 第一阶段目标

第一阶段不追求“像电影一样聪明”，只追求“能稳定帮忙做一件真实的事”。

第一阶段最小版本包括：

- 支持命令行文本对话。
- 支持长期记忆文件，用来记录用户偏好、项目目标、重要事实。
- 支持读取指定目录下的文本文件，并基于文件内容回答问题。
- 支持执行少量白名单工具，例如列目录、读文件、写笔记、生成总结。
- 每次执行动作前留下日志，方便回看它做了什么。

第一阶段暂不做：

- 不做复杂 UI。
- 不做全屋设备控制。
- 不做自动发消息、自动转账、自动删除重要文件。
- 不做“自我进化”或修改底层模型。
- 不做公司项目内嵌，避免个人项目和工作项目混在一起。

## 4. 技术选型

第一阶段使用 Python 3.13 系列作为主语言。

理由：

- Python 更适合快速连接大模型、文件系统、语音、自动化脚本和本地工具。
- 个人助手早期最重要的是快速验证闭环，而不是复杂工程架构。
- 后续如果需要桌面界面，可以再接入 Web 前端、Electron、Tauri 或本地 Web UI。
- Java 适合稳定后端和企业系统，但对个人桌面自动化、AI 工具编排、语音实验来说起步较重。

## 6. 核心模块

### 6.1 对话入口

第一版从命令行开始。用户输入一句话，助手返回回答。后续再扩展语音输入和语音播报。

### 6.1.1 本地确定性大脑

Jarvis Lite 需要先具备不依赖大模型的本地确定性大脑。它负责把常见自然语言映射为内部命令或工具调用，管理长期记忆、知识库、常用目录、最近上下文、日报、更新和桌面动作，并在参数缺失时给出明确提示。

最近上下文负责保存最近资料、最近目录和最近搜索结果，让用户可以连续说“给这个资料打标签”“打开这个目录”“查看第二条结果”“查看最近上下文”。这些轻量状态保存到项目外运行态文件，避免重启后丢失短期工作线索。

本地大脑是语音入口的基础。语音识别最终仍会得到文本，如果文本层只能理解斜杠命令，语音体验也不会自然。

### 6.1.2 大模型外脑

后续接入大模型时，大模型不应直接绕过本地执行层。推荐定位是外部百科、复杂语言理解顾问和多步骤规划器。数据流应是：用户输入 -> 本地意图层判断 -> 本地可处理则直接执行 -> 需要外部知识或复杂规划时再调用大模型 -> 本地大脑决定是否执行工具。

这种分层可以让助手逐步成长：先积累记忆、经验、知识和技能，再考虑是否需要模型微调或本地模型训练。

### 6.2 记忆系统

先用本地 Markdown 文件保存长期记忆，不急着上数据库。

记忆内容包括：

- 用户长期目标。
- 技术偏好。
- 常用项目目录。
- 重要生活偏好。
- 助手行为规则。
- 用户身份和自我介绍。

经验记忆单独保存在 `memory/experiences.md`，用于沉淀可复用流程和表达习惯。它和身份、偏好等长期画像分开，避免 profile 变成混杂流水账。

### 6.3 工具系统

第一版只允许低风险工具：

- 读取指定目录文件。
- 写入项目内笔记。
- 生成总结文档。
- 列出目录结构。
- 记录操作日志。

后续再逐步加入：

- 浏览器搜索。
- 日程管理。
- 本地应用启动。
- 语音控制。
- 家居设备接口。

### 6.4 权限机制

低风险动作可以自动执行，例如读文件、生成总结、写入本项目目录。

高风险动作必须确认，例如：

- 删除文件。
- 覆盖重要文件。
- 调用外部消息接口。
- 控制设备。
- 涉及付款、账号、隐私资料的动作。

## 7. 第一版验收标准

第一版完成时，应满足以下条件：

- 可以在本机启动命令行助手。
- 可以进行多轮文本对话。
- 可以读取 `memory` 中的用户偏好并用于回答。
- 可以写入长期记忆，并基于记忆回答“我是谁”。
- 可以把一段对话总结写入本项目目录。
- 可以读取 `data` 目录中的文本资料并回答问题。
- 所有工具调用都有日志记录。
- 有最基础的本地测试，验证记忆读写和工具白名单逻辑。

## 8. 路线图

### 阶段 1：命令行助手

目标：让助手能通过命令行对话、读写记忆、读文件、写总结。

### 阶段 2：个人知识库

目标：导入 Markdown、txt、PDF 摘要、聊天记录，让助手能围绕个人资料问答。

### 阶段 3：语音入口

目标：加入语音识别和语音播报，让它能像助手一样被口头调用。

### 阶段 4：桌面自动化

目标：打开程序、整理文件、生成日报、辅助写代码、管理常用目录。

### 阶段 5：外部系统接入

目标：接入日程、浏览器、消息、家居或车机等外部系统，但必须保留权限确认。

### 横向阶段：自然语言本地大脑

目标：让常用本地能力不依赖斜杠命令即可触发，为桌面助手、语音入口和后续大模型外脑接入提供统一文本理解层。
